## 1. Purpose of the ML Module inside the Ecosystem
TerraAlert aggregates natural disaster feeds globally. However, raw external telemetry (like news RSS feeds or non-standardized tweets) often lacks structured categorizations. The ML module's purpose is to autonomously parse these unstructured, noisy textual descriptions and classify them into strict, standardized disaster profiles (e.g., `flood`, `earthquake`). This normalization acts as the bedrock for the localized risk decay mathematical calculations.

## 2. Problem Formulation
The core challenge modeled mathematically is a **Natural Language Processing (NLP) Supervised Text Classification** problem. It maps an input vector of text features (tokenized substrings) against a predefined, discrete set of mutually exclusive categorical target labels.

## 3. Type of ML Task
* **Target Operation**: Multi-class text classification.
* **Secondary Operations**: The overarching system also applies deterministic heuristics (Risk Scoring/Decay Calculations), but the pure ML component is strictly bound to text categorization.

## 4. Input Features and Expected Outputs
* **Inputs (X)**: A consolidated text document string. In production inference, this concatenates the report's `title` alongside its paragraph `content` (e.g., `"Flash flood warning issued after heavy rainfall..."`).
* **Outputs (y)**: A single categorical string label referencing the active threat condition (`flood`, `earthquake`, `heatwave`, `cyclone`, or isolated to `unknown` boundaries).

## 5. Dataset Usage and Preprocessing Pipeline
* **Source Data**: Ground-truth samples reside within `ml/datasets/labeled/training_data.csv`. This CSV maintains strict adherence to a `text` and `label` header structure.
* **Training Pipeline Split**: Datasets execute standard `train_test_split()` partitioning arrays 80% to memory generation and 20% dedicated toward validation testing. Splits are explicitly stratified (`stratify=data["label"]`) preventing class imbalances across rare events.
* **Inference Preprocessing (Production)**: Online data pipes heavily through `normalize_text` (found in `app/utils/data_preprocessor.py`). This converts strings explicitly `.lower()`, substitutes `[^a-z0-9\s]` sequences via Regex ensuring punctuation stripping, and collapses all extraneous white-spacing seamlessly normalizing the bounds.

## 6. Feature Engineering
The architecture foregoes dense word-embedding neural logic prioritizing extreme speed and CPU-compatible vectorization.
* **TF-IDF Vectorization**: Textual features are encoded utilizing Scikit-learn's `TfidfVectorizer`.
* **N-Gram Ranges**: Explicitly extracting `(1, 2)` (Unigrams and Bigrams) detecting phrases natively (e.g., "flash flood" explicitly captured).
* **Document Frequencies**: Maps thresholds targeting `min_df=1` ingesting granular datasets inherently flawlessly without dropping niche vocabulary cleanly.

## 7. Model Architecture / Algorithm Choice
**Logistic Regression (`LogisticRegression`)**
* **Rationale**: Bypassing heavy Deep Learning constraints natively relying on classic stochastic linear classification. Logistic Regression was uniquely chosen for sparse TF-IDF text arrays granting incredible prediction speeds (sub-10ms), highly interpretable algorithmic weights, and zero GPU infrastructure deployment dependencies. 
* **Parameters**: Hardcoded to `max_iter=1000` protecting against non-convergence errors during high-dimensionality matrix iterations dynamically.
